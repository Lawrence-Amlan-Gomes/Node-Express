## Structural notes

This is a real Next.js + TypeScript + Tailwind v4 app — the curriculum delivery mechanism (Sidebar, Dashboard, RoadMap page, one page per built topic), mirroring a sibling "Next Js" mentor project's proven shell exactly. **The subject matter is backend-only; the delivery app is not** — don't conflate the two again (this got mis-read once on 2026-07-13; see `co-founder/current-state.md`'s Momentum note).

- Run with `npm run dev` (port 3000 normally, falling back to the next free port only if 3000 is unavailable). Also available: `npm run build`, `npm run typecheck` (`tsc --noEmit`), `npm run lint` (ESLint via `eslint.config.mjs`).
- **Pinned versions, not a mistake**: `next` `^16.2.10`, `react`/`react-dom` `^19.2.7`, `tailwindcss`/`@tailwindcss/postcss` `^4.3.2`, `typescript` `^6.0.3`, `eslint`/`eslint-config-next` `^9.39.4`/`^16.2.10` — copied from the sibling project's own pins, which exist because newer majors (`typescript@7.x`, `eslint@10.x`) broke this Next.js version's build-time typechecking and lint config in this exact environment. Don't bump without re-verifying `npm run build` and `npm run lint` still pass clean.
- `src/data/curriculum.ts` is the single source of truth for the Sidebar nav, the RoadMap page, and routing — Stage A→F, backend-specific (see `co-founder/roadmap.md` for the prose version). `hasPage: true` topics have a real folder at `src/app/topics/<slug>/page.tsx`; everything else falls through to `src/app/topics/[slug]/page.tsx`, a stub. Build order is learner-driven, one topic at a time.
- `src/components/StudyPage.tsx` is the shared template every topic page uses (sections: `paragraphs` → `extra` → `filePointer(s)` → `demo`). Ported near-identically from the sibling project's own `StudyPage.tsx` — same `GuideSection`/`FilePointer`/`ComparisonCard`/`FlowChain`/`CodeBlock`/`StatusBadge` components, same roles.
- **`TerminalFrame.tsx` is the one deliberate departure from the sibling's shell** — it replaces `BrowserFrame` (mock browser chrome around a rendered UI demo) with mock terminal chrome around real captured command output, since most backend concepts have no page to render. A demo's `demoCommand` prop is the literal command that produced the output shown.
- `examples/<Topic>/<SubDemo>/` — standalone, real, runnable mini-projects (own `package.json`; a real `.env` only where something is genuinely configurable). `src/example-runners/<Topic>/<SubDemo>Runner.tsx` proves each one live inside the app, usually as a plain async Server Component that actually executes real code on every render (see `co-founder/build-conventions.md` for the full runner taxonomy and the confirmed Turbopack `execFileSync` gotcha — use `execSync("node file.js", { cwd })`, never `execFileSync("node", [pathString])`, when a runner needs to spawn a process).
- **2026-07-13 note:** sibling projects ("Next Js", then "JavaScript") were briefly nested inside this folder so Claude could read their mentor-system conventions/state for reference, then removed by the user. If either folder is absent, that's expected. `tsconfig.json`/`eslint.config.mjs`/`.gitignore` all defensively exclude both in the meantime. **Standing rule: the moment any new sibling folder gets nested in here temporarily, add it to all three files immediately** — missing this on the `JavaScript/` folder caused 713 stray lint errors from its own `.jsx` files before being caught.

## Dev server / port rules (apply every session, automatically)

- At the start of a new session, start the dev server (`npm run dev`) and tell the user which port it's running on.
- Record the port + start command in `co-founder/ports.md` so future sessions know what "my server" refers to — re-verify with `lsof -i :<port>` rather than trusting a stale recorded PID, since PIDs don't survive a restart.
- **Never kill a port/process you did not start yourself.** If something is already listening on the intended port and it's not something you started this session lineage, don't touch it — pick a different port or ask the user.
- You **may** kill and restart a server **only if you started it** — e.g. after a code change that needs a restart. Tell the user when you do this and what the new port is.
- On "End Today" (see `skillCoFounderMentor.md`), always kill the dev server you started before ending the session, and update `co-founder/ports.md` to reflect it's stopped.

## Framework/structure decisions

- Curriculum delivery: Next.js viewer app, per Structural notes above (decided 2026-07-13, mirroring the sibling project).
- **Backend target: Express 5**, not Express 4 (decided 2026-07-13, research-backed — see `co-founder/roadmap.md`'s research-pass note). Express 5 is npm's install default and the Express Technical Committee's production-recommended release as of 2026; Express 4 enters formal EOL no sooner than October 2026. When Stage B examples get built, use Express 5's actual behavior (e.g. automatic rejected-promise-to-error-middleware forwarding) as the primary taught pattern, with the pre-5 manual try/catch pattern taught only as legacy-codebase awareness, not the default.
- **Backend is TypeScript-first**, not a JS-then-TS afterthought — added as its own early Stage A topic 2026-07-13 after research confirmed TS is now standard for serious Node projects, and because the learner already has verified plain-TypeScript mastery from a sibling project (see `co-founder/learner-profile.md`) ready to apply here.
