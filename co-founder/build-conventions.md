# Build Conventions

Owned by `skillCoFounderMentor.md`. **Read this before building or editing any topic page, example, or runner.** These rules exist because the sibling "Next Js" project (same user, different mentor instance) shows a repeating, confirmed pattern: any drift between what a demo claims to show and what it actually does gets caught, not glossed over. Assume the same scrutiny applies here.

## The shape of this app

This project has a real Next.js viewer app (`src/app/`, `src/components/`, `src/data/curriculum.ts`) that delivers the curriculum — Sidebar, Dashboard, RoadMap page, and one `page.tsx` per built topic via `StudyPage` — mirroring the sibling project's shell exactly (same component names/roles where the concept transfers directly: `GuideSection`, `FilePointer`, `ComparisonCard`, `FlowChain`, `CodeBlock`, `StatusBadge`). **The one deliberate departure:** the sibling's `BrowserFrame` (proving a UI demo by rendering it in mock browser chrome) doesn't fit here, since the subject matter is backend — there's no page to render for most topics. `TerminalFrame` (`src/components/TerminalFrame.tsx`) is the replacement: mock terminal chrome showing the real command that was run and its real captured output. A `StudySection.demoCommand` there is the equivalent of the sibling's `demoTabTitle`.

Each topic gets, once actually reached (build order is learner-driven, one topic at a time — never front-load a topic not yet started):

- `examples/<Topic>/<SubDemo>/` — a standalone, **real, runnable** mini-project (own `package.json`, and a real `.env` only where the example genuinely has something configurable — don't add a decorative one just for shape). Genuinely runnable on its own (`npm start` or equivalent), not a fragment that only makes sense glued into something bigger.
- `src/example-runners/<Topic>/<SubDemo>Runner.tsx` — what actually proves the example live inside the app. Two real patterns, matching the sibling project's own split:
  1. **Plain async Server Component that executes real code every render**, no client-interactivity needed — e.g. spawning the real example script/server and capturing real output (see "The Turbopack execFileSync gotcha" below), or importing and calling a real exported function directly. Zero faking; this is the default and should be preferred whenever the concept allows it.
  2. **A "use client" component with real local state**, only when something about the concept can't work executed straight inside this app's own render (needs its own real routing/params/thrown error that would conflict with this app's own). Rare for backend topics compared to the sibling's frontend ones — most backend concepts (event loop, middleware, routing, error handling) CAN be proven with a real, live, server-side execution instead of a rebuilt stand-in. Reach for this only when pattern 1 genuinely doesn't apply.

## The Turbopack execFileSync gotcha (confirmed 2026-07-13)

Spawning a real example process from a Server Component runner via `child_process.execFileSync("node", [somePathString])` **fails `next build`** with `Module not found ... server relative imports are not implemented yet` — Turbopack's build-time file tracer statically treats a literal/computed string in that array-argument position as a module specifier to resolve, regardless of whether the path is built with `path.join`, `__dirname`, `process.cwd()` concatenation, or even a bare relative filename. Confirmed directly across all of those variations — every one broke the build the same way.

**The fix that works:** use `child_process.execSync("node event-loop-order.js", { cwd: ..., encoding: "utf-8" })` — a single shell-command string plus a `cwd` option, not an array of path-shaped arguments. This sidesteps whatever heuristic Turbopack applies to `execFileSync`'s argument array. See `src/example-runners/NodeRuntimeFoundations/EventLoopOrderRunner.tsx` for the working reference implementation. Use this pattern for every future runner that needs to spawn a real process rather than rediscovering the same failure.

## The missing body text-color gotcha (confirmed 2026-07-13)

`src/app/globals.css` styled `body` with a font-family but never a default `color` — so any text node without an explicit Tailwind `text-*` class (title/body/sublabel/muted, or a semantic accent) fell back to the browser's default black, invisible against this app's black background. Caught in the `LibuvPhasesDiagram` on the `what-is-nodejs` page, where a raw `{i + 1}.` and ` — ` sat directly in a `<div>` with no color class. Fixed globally by adding `color: var(--color-body);` to the `body` rule — this is the actual root-cause fix, not just patching the one diagram. **When writing any new bespoke diagram, every text-bearing element still needs an explicit color class anyway** (don't rely on inheritance as the primary plan) — the body-level fix is a safety net, not a reason to skip explicit `text-body`/`text-muted`/accent classes on new markup.

## One mini-project PER SECTION, not one shared per topic (confirmed 2026-07-14)

Every distinct `StudySection` on a topic page needs its OWN separate, self-contained `examples/<Topic>/<SectionSpecificName>/` mini-project — its own folder, own `package.json`, own server/script file containing ONLY the code that section actually demonstrates. **Do not** build one shared server.js (or one shared example project) and then split it across multiple sections via multiple demo scripts that each hit different routes on the same file — that was the actual mistake made across all three of Stage B's first topics (caught directly by the user on "The Middleware Pipeline," where 3 sections covering 3 genuinely unrelated concepts — pipeline order, an auth-check that only works in the right position, a forgotten next() hanging a request — all lived in one shared `server.js`).

**Why this matters, beyond just tidiness:** a shared file across sections makes it unclear which code belongs to which lesson, and edits/fixes to one section's demo risk silently affecting another section's. Each section is a separate lesson and deserves a separate, fully self-contained mini-project, even if that means near-identical boilerplate (a `package.json`, an Express import, a `pathToFileURL` listen-guard) gets repeated across a topic's several mini-projects — that repetition is the correct trade-off here, not premature duplication to avoid.

**How to apply going forward:** when planning a new topic, decide the sections FIRST, then build one mini-project per section from the start — never build one broad example and carve demo scripts out of it after the fact.

## FilePointers must stay inside that topic's own `examples/` folder (confirmed 2026-07-13)

Every `filePointer`/`filePointers` entry on a topic page must resolve to a path inside that topic's own `examples/<Topic>/<SubDemo>/` mini-project — **never** to a root-level file (root `package.json`, `package-lock.json`, `CLAUDE.md`, etc.), even when the root file is real and the claim being made is true. The teaching surface for every topic is its own self-contained mini-project, full stop. If a concept needs a real artifact to point at (a real dependency to show semver ranges, a real generated lockfile), build/install that artifact inside the example folder itself — e.g. `examples/ModulesNpmPackageJson/CjsEsmInterop/package.json` declares a real dependency (`picocolors`, actually used in `main.js` to color its real output) specifically so its own `package-lock.json` could be pointed at directly, instead of reusing the root project's. Caught directly by the user on the "Modules, npm & package.json" topic pointing at root `package.json`/`package-lock.json` — don't rediscover this.

## Root tooling must exclude `examples/` (confirmed 2026-07-13)

Root `tsconfig.json`'s `include: ["**/*.ts", "**/*.tsx", ...]` and `eslint.config.mjs` both glob broadly by default — the first example with real `.ts` source files (`examples/TypeScriptWithNode/TypedBackendSetup/`) got pulled straight into the root app's own typecheck (under the ROOT app's `moduleResolution: "bundler"` settings, not that example's own `nodenext` + `rewriteRelativeImportExtensions` config), producing false errors that had nothing to do with the app itself. Fixed by adding `"examples"` to root `tsconfig.json`'s `exclude` array and `"examples/**"` to `eslint.config.mjs`'s `ignores`, alongside the existing sibling-project exclusions — both already fixed, so this shouldn't recur, but if a *new* root tool ever gets added (a second linter, a different type-checker config, etc.), remember every `examples/<Topic>/<SubDemo>/` mini-project is self-contained with its own tooling and must stay out of the root app's scan.

## The same-process blocking-demo gotcha (confirmed 2026-07-14)

Proving "blocking the event loop freezes the whole server" with a single Node process acting as BOTH the HTTP client and the server (fetch() calling into its own http.createServer) has two real, confirmed traps:

1. **Racing two different routes (fire /blocking and /ping "concurrently" via Promise.all, measure which one is slow) is genuinely flaky** — whichever request's TCP connection happens to get accepted and dispatched to the JS callback first isn't guaranteed to be the one issued first in code. Confirmed directly: out of 3 runs, one showed /ping resolving in 8ms even while /blocking's busy-loop should have still been running.
2. **A client-side setTimeout "head start" to force ordering doesn't work either** — since client and server code run on the exact same single JS thread in a same-process demo, a client-side timer scheduled to fire in 30ms can itself get swallowed by the server's synchronous busy-loop if that loop starts running before the 30ms elapses (confirmed via added instrumentation: a timer set for 30ms didn't fire until t+316ms, right after a 300ms busy-loop that started at t+14ms finished — the timer was blocked too, exactly like everything else).

**The fix that works, deterministically, across 8+ runs:** don't race two different routes — fire TWO CONCURRENT requests to the SAME route and measure the real TOTAL time for both. Blocking work can't overlap with itself (total ≈ 2x the per-request work, serialized); non-blocking work does overlap (total ≈ 1x). This sidesteps the ordering race entirely, since it no longer matters which request's connection gets dispatched first. See `examples/AsyncPatternsOnServer/BlockingVsNonBlocking/blocking-server-demo.js` for the working reference implementation (moved 2026-07-17 out of a shared `AsyncOnAServer` folder into its own dedicated mini-project, per the one-mini-project-per-section rule below).

## The ESM "run directly vs imported" check (confirmed 2026-07-14)

Every Express example server needs to both (a) export its `app` for a demo script to import and drive via ephemeral-port `fetch()` calls, and (b) actually call `.listen()` for real when run directly via `node server.js`. The standard CommonJS check (`require.main === module`) doesn't exist in ESM — the naive-looking ESM equivalent, comparing `import.meta.url` against `` `file://${process.argv[1]}` ``, is a real, confirmed bug: `process.argv[1]` is frequently a RELATIVE path (e.g. `"server.js"`), while `import.meta.url` is always an absolute `file://` URL, so the string comparison silently never matches — `.listen()` never runs, the process exits immediately with zero output, and it looks like nothing is wrong until you actually try to hit the server (confirmed directly: this exact bug shipped once in `examples/ExpressAppRouting/BasicExpressApp/server.js` before being caught).

**The real fix:** `import { pathToFileURL } from "node:url"; if (import.meta.url === pathToFileURL(process.argv[1]).href) { app.listen(...) }` — `pathToFileURL` correctly resolves a relative-or-absolute `argv[1]` into the same absolute-URL shape as `import.meta.url`, regardless of how the script was invoked. Use this pattern for every future Express server example that needs both a real standalone `node server.js` entry point AND an importable `app` for a demo script.

## The Express 4 async footgun crashes the process, it doesn't just hang (confirmed 2026-07-15)

Conventional wisdom (and this project's own earlier curriculum summary) described the pre-Express-5 async-handler footgun as "the request hangs forever." Verified for real on the pinned Node version in this project (Node v22.22.3): an Express 4 async route handler that throws, with no try/catch and no next(err), does NOT just hang that one request — it produces an **unhandled promise rejection**, and modern Node.js (v15+) treats that as a fatal error by default, **crashing the entire process**. Confirmed deterministically across multiple runs: `spawn`ing a real Express 4 server as a separate process and hitting the offending route ends with that process exiting with code 1 (a `fetch failed: other side closed` `SocketError` on the client side, from the connection being yanked out from under it) — taking down every other in-flight request on that same server, not just the one that triggered it. See `examples/ErrorHandlingExpress/LegacyTryCatch/demo.js` for the working reference implementation (it spawns the server as a real child process specifically so there's a real process to observe dying, rather than trying to survive the crash in-process).

**Why this matters pedagogically:** this is a stronger, more accurate teaching point than "hangs forever" — Express 5's automatic rejected-promise-forwarding isn't just closing a request-hanging edge case, it's preventing one bad async route from taking the ENTIRE server down. Use "crashes the whole process" as the framing going forward, not "hangs," whenever this topic comes up again.

## Wrong/anti-pattern code lives only in comments, never as real running code (confirmed 2026-07-15)

Caught directly by the user across three shipped Stage B examples (`MiddlewarePipeline/AuthOrderMatters`'s `/protected-wrong` route, `MiddlewarePipeline/ForgottenNext`'s entire premise, `ErrorHandlingExpress/LegacyTryCatch`'s `/no-trycatch` route): shipping a deliberately-broken/anti-pattern route as REAL, executing code in an example's `server.js` is wrong, even when the point of the section is to teach why that pattern is broken. The reasoning: every example's `server.js` doubles as reference material for code the user might genuinely reuse or pattern-match against when writing real production code later. A file that contains real, running "wrong" code — even clearly commented as wrong — risks becoming muscle memory precisely because it's real, syntactically valid, executed code sitting right next to the correct version.

**The fix, applied as a standing rule for every future topic, not just a one-time cleanup:** when a section's whole teaching point is "here's a mistake and why it's bad," the wrong/anti-pattern version is written ONLY as a comment block inside the file — clearly labeled (e.g. "THE MISTAKE, shown only as a comment, never actually run in this project") — never as a real route, real middleware, or any other code that Express/Node actually executes. The comment should still state the real, verified consequence of that mistake (crashes the process, hangs forever, silently does nothing) as a plain factual statement, because that fact was actually verified by running the broken version once during development — it just doesn't ship as part of the example. The ONLY code that actually runs in the shipped example is the correct, production-worthy version — the one the user should treat as "what I'd actually write." Demo scripts only ever exercise the correct, real, running routes.

**Verification discipline still applies to the comment's claims** — before writing a comment asserting what the broken version does (hangs / crashes / silently succeeds), actually run it once, off to the side, and confirm the real behavior first. Don't write an unverified claim into a comment just because it's "only a comment" now.

**This also covers incompleteness, not just active bugs** — caught a 4th instance the same session, on `ErrorHandlingExpress/AutomaticForwarding`: the shipped `/risky` route threw with no try/catch (correct — that's the point of the section) but had NO error-handling middleware registered at all, silently relying on Express's own default handler (which leaks a raw stack trace to the client). That's not a route a real backend dev would ship either, even though nothing about it looks like an obvious "bug" the way a forgotten `next()` does. Fixed by adding a real error middleware to the shipped code, with the default-handler fallback behavior (verified: a full HTML page with the stack trace, since `NODE_ENV` isn't `"production"`) described only in a comment. **When reviewing a section's shipped code against this rule, check for what's silently MISSING (a middleware nobody registered, a check nobody wrote) just as much as for what's ACTIVELY wrong** — both count as "not what a real backend dev would ship."

## The database environment decision, resolved for real (confirmed 2026-07-15)

The blocker recorded above (no local Postgres server, no MongoDB, no Docker) got resolved by a better option the user already had: real REMOTE database instances, not local ones. The user has a real PostgreSQL server running on Coolify (self-hosted) and a real MongoDB Atlas cluster, both already used for other real projects — connection strings supplied directly, verified working immediately (`psql` reached Postgres 16.14; the `mongodb` driver reached Atlas and listed the cluster's existing databases).

**Both are genuinely shared infrastructure with other real data on them** — confirmed directly before writing anything: Postgres's `public` schema already had real tables (`clicks`, `links`, `users`, apparently a URL-shortener app); the Atlas cluster already had ~14 other real databases, including an existing `express` database with one empty `users` collection. **The isolation convention for this project, going forward:**
- **Postgres:** every example uses a dedicated schema, `node_express_learning` (via `?schema=node_express_learning` on the connection string — Prisma's own documented mechanism for targeting a non-public schema without needing the multiSchema preview feature). Never touch `public`.
- **MongoDB:** every example uses the `express` database (the path segment on the connection string) and a dedicated collection name, `learning_tasks` — deliberately distinct from that database's pre-existing empty `users` collection so there's no chance of a future collision if that collection ever gets populated by whatever it was originally for.
- Every example's CRUD demo cleans up fully after itself (delete-all at the start, delete the created row/document at the end) specifically because these are real shared servers, not throwaway local ones — a demo re-run on every Next.js render must never accumulate garbage there.

This is also a genuinely better teaching setup than local Docker/Homebrew would have been: connecting to a real remote database via a connection string, and having to actively namespace your own work to avoid colliding with someone else's data on a shared server, is closer to real production/staging-environment experience than a local single-tenant database ever would be.

**Correction, confirmed 2026-07-16 — this "reuse the same schema across topics" rule was WRONG for Prisma Migrate specifically.** Tried to run `prisma migrate dev` for the "Migrations & Schema Design" topic against this same `node_express_learning` schema (to add a new `Post` table alongside the existing `db push`-created `Task` table) — Prisma immediately detected the pre-existing `Task` table as **drift** (since it has no migration history behind it) and demanded a full schema RESET to reconcile, which would have dropped `Task` too. It aborted non-interactively rather than proceeding (no data lost), but the underlying lesson is real and important: **`prisma migrate dev` needs to fully own a schema via its own migration history — never point it at a schema that already has non-Migrate-managed tables (from `db push`, raw SQL, or anything else) in it.**

**Corrected standing rule:** every topic that uses `prisma migrate dev` gets its OWN dedicated Postgres schema (e.g. `node_express_learning_migrations` for that topic), separate from `node_express_learning` (which stays the `db push`-only schema from "Connecting Real Databases"). MongoDB has no equivalent constraint — its database/collection reuse across topics is still fine, since Mongoose never tracks or enforces schema history the way Prisma Migrate does. This mixing-db-push-and-migrate-in-one-schema failure is itself genuinely good, verified teaching content for the migrations topic (a real reason production teams pick ONE schema-management approach per database and stick to it).

## The dotenv 17.x random startup banner (confirmed 2026-07-15)

`dotenv@17.4.2`'s `.config()` call prints a random one-line "tip" banner to stdout on every call (e.g. `◇ injected env (1) from .env // tip: ⌘ enable debugging`) unless silenced — confirmed to vary across runs (different random tip each time), which would otherwise leak non-deterministic noise into a demo's captured output. **Fix: pass `{ quiet: true }` to `.config()`** — e.g. `require("dotenv").config({ quiet: true })`. Apply this on every future example that uses dotenv's `.config()` call directly (import-style `"dotenv/config"` usage, as in the earlier `EnvConfigWithDotenv` example, did not show the banner under the conditions tested — but default to `{ quiet: true }` on any explicit `.config()` call to be safe).

## Never let two mini-projects' demos share the literal same real table/collection (confirmed 2026-07-16, two real incidents in one session)

The "Connecting Real Databases" topic's `CrudSideBySide` mini-project was originally built to deliberately reuse the exact same real Postgres `Task` table and the exact same real Mongo `learning_tasks` collection as the topic's other two mini-projects (`PostgresWithPrisma`/`MongoWithMongoose`) — reasoned at the time as "proving the same operations against the same real resource." This was wrong, for a concrete, confirmed reason:

**Incident 1 — a real production build failure.** `npm run build` failed prerendering `/topics/orm-query-builder` with `Cannot read properties of null (reading 'toObject')`. Root cause: Next.js renders a page's async Server Components (this app's runners) CONCURRENTLY, not one at a time. `MongoWithMongoose`'s demo.js and `CrudSideBySide`'s demo.js were therefore both actually running at the same real moment, both against the same real `learning_tasks` collection — one's `deleteMany()` cleanup step wiped out a document the other was still mid-`findByIdAndUpdate`/`findByIdAndDelete` on, so that call returned `null` and the demo crashed. The equivalent risk existed on the Postgres side too (both used the literal same `Task` table), it just hadn't been triggered yet by the time this was caught.

**Incident 2 — fixing it caused a second, worse near-miss.** The fix was to give `CrudSideBySide` its own dedicated table (`@@map("SideBySideTask")` in its own separate `schema.prisma`) and its own dedicated Mongo collection (`learning_tasks_side_by_side`). But running `prisma db push` from `CrudSideBySide`'s own folder to create that new table **silently RENAMED the existing shared `Task` table to `SideBySideTask` instead of creating a new one** — with no warning, no prompt, nothing in the output hinting at it. Cause: `prisma db push` diffs against the database's ACTUAL live schema state (not just "this project's own" objects), and since `CrudSideBySide`'s schema no longer declared any model mapped to `Task`, while declaring one mapped to `SideBySideTask` with an IDENTICAL column shape to the existing `Task` table, Prisma's diff engine concluded "rename" was the minimal-diff operation — and just did it. This broke `PostgresWithPrisma`'s demo immediately (`the table node_express_learning.Task does not exist`). Fixed by: renaming the table back via raw SQL (`ALTER TABLE ... RENAME TO "Task"`, plus renaming the leftover `SideBySideTask_pkey`/`SideBySideTask_id_seq` constraint/sequence names that don't auto-follow a table rename), then creating the genuinely separate `SideBySideTask` table via a plain SQL `CREATE TABLE` (not `db push`), then running `prisma generate` only (never `db push`/`migrate` again) against the now-correct, pre-existing table.

**Standing rules, both confirmed the hard way:**
1. **Every mini-project's demo gets its OWN dedicated real table/collection, even within the same topic, even when two mini-projects are proving "the same operation."** Sharing real storage between two independently-executed scripts is a genuine concurrency bug on this app's own rendering model, not just a style question.
2. **Running `prisma db push` (or any schema-modifying Prisma command) from a second, unrelated Prisma project against a schema that ALREADY has real tables from a different mini-project is dangerous** — Prisma's diffing engine operates on the database's actual live state, and can rename/alter a table it doesn't "own" if the shape happens to match closely enough, with zero warning. When a new table is needed in a schema that already has other real tables, prefer creating it via a plain, explicit SQL `CREATE TABLE` (verified to only affect the one new object) over `db push`, then run `prisma generate` (not `push`) once the table already exists matching the schema.

**Correction, confirmed 2026-07-16 (again), while building "Transactions & Query Performance" — rule 2 above UNDERSTATED the real danger.** It isn't just a rename risk when table shapes happen to coincide. Ran `prisma db push` for a brand-new `PostgresNPlusOne` mini-project (model shapes: `Author`/`Post`, genuinely nothing like the existing `Account` table) against the SAME schema an earlier mini-project (`PostgresTransactions`) had already pushed its `Account` table into — and `db push` silently DROPPED `Account` entirely, because it treats the CURRENT project's `schema.prisma` as the complete, authoritative desired state for the WHOLE target schema, and deletes anything in the live schema it doesn't declare. No shape collision needed to trigger this — any undeclared table in the target schema is at risk, every time. Recovered via raw SQL `CREATE TABLE` matching the original shape exactly.

**Corrected standing rule, superseding rule 2 above:** **every mini-project that uses `prisma db push` or `prisma migrate dev` gets its own dedicated Postgres schema, full stop — never share a schema across two independent Prisma projects, regardless of whether table names or shapes overlap.** (Two mini-projects MAY still safely share a schema if only ONE of them ever runs a Prisma schema-sync command against it and the other only reads/writes via already-generated queries — but that's fragile and not worth relying on; a fresh dedicated schema per mini-project is simpler and categorically safe.) This is now the confirmed convention across three real topics: `node_express_learning` (Connecting Real Databases), `node_express_learning_migrations` (Migrations & Schema Design), `node_express_learning_transactions` and `node_express_learning_nplusone` (Transactions & Query Performance, two schemas for two Postgres mini-projects in the same topic). MongoDB has no equivalent risk — Mongoose never syncs or diffs a database's full collection list the way Prisma does, so collection reuse across topics (when not a concurrency risk per rule 1) remains fine.

## When the topic is genuinely browser-enforced, use a real headless browser (confirmed 2026-07-16)

CORS turned out to be enforced ENTIRELY by the browser, not by Node, not by Express, not even by Node's own `fetch()` — verified directly before building anything: a plain Node script's `fetch()` call sends no preflight OPTIONS request and is blocked by nothing, even hitting a real API that sends zero CORS headers back (a real 200 comes through fine). Writing a Node-only demo that "prints the CORS headers" and asserts blocking would have been fabricating behavior that literally cannot happen outside a real browser — a violation of the "never fabricate behavior" rule, not just a shortcut.

**The fix: use Playwright (a real headless Chromium browser) for both of this topic's mini-projects** (`examples/CorsFrontendHandshake/PreflightVsSimple`, `examples/CorsFrontendHandshake/BrowserEnforcedBlocking`) — a real two-origin setup (two Express servers on two different ports = two real different origins), with a real page's own `<script>` making real cross-origin `fetch()` calls, captured via Playwright's `page.evaluate()` and `page.on("console")`. This produced genuinely real evidence: the exact real Chromium console error text (`"has been blocked by CORS policy: ..."`), the real request order proving preflight only fires for non-simple requests, and the real credentials opt-in behavior (a cookie already stored for an origin is NOT sent cross-origin unless the client's fetch call ALSO passes `credentials: 'include'`).

**Practical notes for reusing this pattern:** `playwright` installs fast as an npm package, but needs a `postinstall: "playwright install chromium"` script to fetch the actual browser binary — confirmed this downloads once and is cached globally (`~/Library/Caches/ms-playwright` on macOS), so every subsequent mini-project's install of the same browser is fast, not a repeated multi-hundred-MB download. Total real execution time for a Playwright-based demo (launch browser, navigate, evaluate, close) was ~0.7s — no real build-time cost concern for the runner pattern.

**Standing rule going forward:** before building any future topic, ask whether the real behavior being taught is actually enforced by Node/Express itself, or by something else entirely (a browser, an OS-level mechanism, a client library) — if it's the latter, a plain Node script proving it is not real evidence, and reaching for the real enforcing environment (a real headless browser, here) is the correct move, not a shortcut to avoid.

## The Jest-writes-to-stderr execSync gotcha (confirmed 2026-07-16)

`execSync("npm test", { encoding, cwd })` in a runner silently captured ONLY the npm preamble ("> pkg@0.1.0 test\n> node ... jest ...") and NOTHING else — the real PASS/✓/summary lines were completely missing from the rendered page, even though running the exact same command by hand in a terminal (with `2>&1`) showed full output. Root cause: Jest's default reporter writes its real human-readable report to **stderr**, not stdout (stdout is reserved for machine-readable output like `--json`) — and `execSync` only returns the child's stdout by default, silently dropping stderr. Confirmed directly: building/curling the `testing-express` page showed the npm preamble with nothing after it, until the command was changed.

**The fix:** append `2>&1` to the command string passed to `execSync` (e.g. `execSync("npm test 2>&1", { encoding, cwd })`) — since `execSync` runs through a real shell by default, this redirection works exactly like it would in a terminal, merging stderr into the captured stdout. Apply this to any future runner that spawns a tool (Jest, or anything else) known to write its main human-readable output to stderr rather than stdout — don't assume every real CLI tool's important output lands on stdout.

## Non-negotiable rules

- **Never fabricate behavior.** If a topic makes a claim about how Node/Express/a library behaves (error propagation, middleware order, a status code, a header, an ordering guarantee), verify it by actually running the code first — don't assert from memory or training data. Library/runtime behavior changes across versions, and this project already found one genuinely surprising, verified-for-real gotcha in its first topic (setTimeout vs setImmediate ordering at the top level of a script is NOT guaranteed — see the `what-is-nodejs` topic page).
- **No fake stand-ins where the topic is genuinely about a real tool.** If the topic is about validation, use a real validator; if it's about a database, connect to a real one (can be local/throwaway, but real). Don't simulate what a real dependency would do.
- **Runner output must match the example it claims to demonstrate, exactly.** If the example prints/returns N things, the runner's shown output must show all N — no silently trimming, no adding anything that isn't really there.
- **Keep pinned-version/gotcha notes in `CLAUDE.md` up to date** if a dependency version or tooling behavior turns out to break something in this environment — don't silently work around it without recording why, the way the Turbopack gotcha above is now recorded.

**A second real mistake caught the SAME session, immediately after the fix above shipped:** the exhaustive rewrite had started appending instructional context (which header to send, "this is request 6 of 7," which port a step belongs to) directly onto the end of `expectBody` strings — e.g. `expectBody: '{"error":"..."} — request 7, still blocked — wait 60 real seconds to reset.'`. - `headers?: Record<string, string>` — real request headers (e.g. `{ "X-User-Id": "1" }`), rendered as their own block under the request URL, before Body.
- `note?: string` — a short, italicized, request-side instruction/context line (e.g. "Send with NO X-User-Id header at all," "Request 6, same 60-second window"), rendered after Body/Headers but still clearly BEFORE the green "Expect:" line.

## `demo.js` only calls a real API — it never talks to an ORM/ODM directly (confirmed 2026-07-17)

Caught directly by the user, across all of Stage C's real-database topics: every mini-project in "Connecting Real Databases," "Migrations & Schema Design," and "Transactions & Query Performance" had `demo.js` importing Prisma/Mongoose and calling `prisma.task.create()` / `Task.create()` etc. straight from a `main()` function. His framing: **"I will not write demo.js in the production while working as a backend developer. demo.js is only for calling the APIs and showing the output, that's it."** — a real, correct call-out that this violated the project's own established pattern (`Project Structure & Config`'s routes/controllers split) everywhere else in Stage B.

**The fix, now a standing rule for any topic whose example genuinely represents a real backend resource (a database-backed CRUD resource, a transfer/transaction endpoint, anything a real client would call over HTTP):** every such mini-project gets the full layering, same as any other Express topic —
- `server.js` — creates the app, wires global middleware, mounts routers, listens on a real fixed port when run directly (same ESM-guard/CommonJS `require.main === module` pattern as every other topic).
- `routes/<resource>.routes.js` — declares path+method → controller function. Never imports Prisma/Mongoose.
- `controllers/<resource>.controller.js` — the ONLY file that imports and calls the ORM/ODM. Real business logic (including things like Prisma's `$transaction` or a real query-count instrumentation counter) lives here, and gets returned as real response data when relevant (e.g. `{ posts, queryCount }`), not printed to console from inside the controller.
- `demo.js` — imports `app` from `server.js`, starts it on an ephemeral port (same pattern as every Stage B demo), then ONLY makes real `fetch()` calls against real routes and prints what comes back. No `require("@prisma/client")` or `require("mongoose")` anywhere in this file.

**Scope note, confirmed directly when asked:** this applies only to mini-projects that represent a real backend/API concern connecting to a real production-shape database. "SQL vs NoSQL" (Stage C topic 1) was explicitly EXEMPTED — it's a pure conceptual data-shape comparison using in-memory `node:sqlite` and plain JS objects, with no real API being taught, so it correctly stays as plain comparison scripts (same reasoning as Stage A's non-server examples).

**A real regression surfaced while doing this, worth flagging as its own lesson:** 3 of the 8 restructured mini-projects (`PostgresTransactions`, `PostgresNPlusOne`, `MongoTransactions`) originally had a final cleanup step in `demo.js` (`await prisma.account.deleteMany()` etc., leaving the real shared table/collection empty when the script finished) — the first restructuring pass dropped that final cleanup by accident, since the new `demo.js` only called a `/reset` endpoint at the START of the run, not a cleanup at the end. Confirmed directly by querying the real Postgres/Mongo state after a run: real rows/documents were left behind. **Fixed by adding a real `DELETE /<resource>` endpoint (clears with no reseed) to each controller, and calling it at the end of `demo.js`, matching the original cleanup behavior exactly.** When restructuring an existing demo.js into an API-calling one, diff its old start-of-file AND end-of-file cleanup behavior against the new version — it's easy to preserve the start (seeding) and silently drop the end (teardown).

**Mongoose connection lifecycle gotcha, confirmed the same session:** `server.js` opens the real `mongoose.connect(...)` once, meant to live for the whole server's lifetime. A `demo.js` that only calls `server.close()` (closing the HTTP listener) does NOT close that Mongo connection — the process hangs forever since Mongoose's open socket keeps Node's event loop alive. Since `demo.js` isn't supposed to import Mongoose at all anymore (see above), it can't call `mongoose.disconnect()` itself either. **The fix: `demo.js` calls `process.exit(process.exitCode ?? 0)` in a final `.finally()`** — a legitimate pattern for a genuinely one-shot script that doesn't own the long-lived connection it's exercising. Postgres/Prisma-backed `demo.js` files did NOT hit this problem (Prisma's connection handling didn't block process exit the same way), so this fix is Mongoose-specific.

Caught directly by the user, the same session as the two gotchas above: three of "REST Conventions & Validation"'s four mini-projects (`StatusCodesVersioning`, `IdempotencyKeys`, `ZodValidation`) had all their route logic written straight inline in `server.js` — no `routes/` or `controllers/` folder at all — even though "Project Structure & Config" (a Stage B topic, built and mastered BEFORE this one) exists specifically to teach that split, and every Stage C database topic already applied it. His framing: this curriculum is meant to be **cumulative**, the same way a real backend job works — once a convention is taught, every later topic's example code should keep using it, not regress back to an earlier, simpler style just because the NEW concept being taught (status codes, idempotency, zod) isn't itself about project structure.

**The standing rule, going forward, for every future topic's example code:** once a structural/organizational convention has been taught (routes/controllers split, layered validation, whatever comes next), treat it as a permanent floor for every subsequent real Express mini-project — regardless of topic — not something that only applies when the topic happens to be "about" that convention. `demo.js`-only-calls-a-real-API (this file's earlier entry) was the first instance of this same principle for database topics specifically; this is the general form of it, covering ALL structural conventions, starting from whichever topic first taught each one.

**How this was fixed on the three affected mini-projects:** each got a real `routes/<resource>.routes.js` (declares path+method → controller function, zero logic) and `controllers/<resource>.controller.js` (the actual handler logic, the in-memory data, the one file that would import a real ORM if this topic used one) — `server.js` shrank down to just creating the app, wiring `express.json()`, mounting the router, and the ESM listen-guard. `demo.js` needed NO changes in any of the three, since it was already `fetch()`-only — proof that getting the `demo.js`-calls-API rule right in the first pass doesn't guarantee the routes/controllers rule was also applied; both need to be checked independently on every new mini-project.

## The routes/controllers layering rule reaches Jest/Supertest examples cleanly (confirmed 2026-07-18)

Rebuilding "Testing an Express App" (Stage E topic 1) confirmed the cumulative layering rule (see above) applies just as cleanly to test-focused topics as to any other real Express example — `IntegrationTestingWithSupertest` and `TestingErrorCases` both got real `routes/`/`controllers/` splits, and neither `server.test.js` needed a single line changed, since both still `import { app } from "./server.js"` exactly as before. Layering the app under test doesn't change the test's own interface at all — a good sign the split is genuinely just organizational, not leaking into behavior.

**`UnitTestingPureFunctions` stayed a plain, non-Express module — correctly, not an oversight.** Its whole teaching point is testing ONE pure function in isolation, with no server at all; forcing it into an Express API would contradict the very contrast (unit vs. integration) the topic exists to draw. This is a genuine, deliberate exemption from the "give every library its own API" rule, not a case that was missed — the same reasoning as "SQL vs NoSQL"'s exemption from the earlier database-layering rule.

**A real port collision was also caught here**: `TestingErrorCases` had been assigned 4061 before `SecurityRateLimiting/BrokenAccessControl` later claimed the same number — moved to 4077 (the next free port after a full audit of every `server.js`'s `PORT ?? N` line). **Standing habit going forward:** run `grep -rhoE "process\.env\.PORT \?\? [0-9]+" examples --include=server.js | grep -oE "[0-9]+" | sort -n` before assigning ANY new fixed port — don't just increment from memory, since Stage E topics were built chronologically before some Stage D topics that ended up reusing the same low numbers.

## Verification discipline

Before writing prose that asserts "Express does X" or "Node does Y" as fact, actually run it and look at the real output first — including running it multiple times when a claim is about ordering/timing, since that's exactly the kind of claim that looks deterministic until you check (see the `what-is-nodejs` topic). Record confirmed gotchas here or in `CLAUDE.md` as they're found, not just in the moment they're discovered.
