# Build Conventions

Owned by `skillCoFounderMentor.md`. **Read this before building or editing any topic page, example, or runner.** These rules exist because the sibling "Next Js" project (same user, different mentor instance) shows a repeating, confirmed pattern: any drift between what a demo claims to show and what it actually does gets caught, not glossed over. Assume the same scrutiny applies here.

## The shape of this app

This project has a real Next.js viewer app (`src/app/`, `src/components/`, `src/data/curriculum.ts`) that delivers the curriculum ‚Äî Sidebar, Dashboard, RoadMap page, and one `page.tsx` per built topic via `StudyPage` ‚Äî mirroring the sibling project's shell exactly (same component names/roles where the concept transfers directly: `GuideSection`, `FilePointer`, `ComparisonCard`, `FlowChain`, `CodeBlock`, `StatusBadge`). **The one deliberate departure:** the sibling's `BrowserFrame` (proving a UI demo by rendering it in mock browser chrome) doesn't fit here, since the subject matter is backend ‚Äî there's no page to render for most topics. `TerminalFrame` (`src/components/TerminalFrame.tsx`) is the replacement: mock terminal chrome showing the real command that was run and its real captured output. A `StudySection.demoCommand` there is the equivalent of the sibling's `demoTabTitle`.

Each topic gets, once actually reached (build order is learner-driven, one topic at a time ‚Äî never front-load a topic not yet started):

- `examples/<Topic>/<SubDemo>/` ‚Äî a standalone, **real, runnable** mini-project (own `package.json`, and a real `.env` only where the example genuinely has something configurable ‚Äî don't add a decorative one just for shape). Genuinely runnable on its own (`npm start` or equivalent), not a fragment that only makes sense glued into something bigger.
- `src/example-runners/<Topic>/<SubDemo>Runner.tsx` ‚Äî what actually proves the example live inside the app. Two real patterns, matching the sibling project's own split:
  1. **Plain async Server Component that executes real code every render**, no client-interactivity needed ‚Äî e.g. spawning the real example script/server and capturing real output (see "The Turbopack execFileSync gotcha" below), or importing and calling a real exported function directly. Zero faking; this is the default and should be preferred whenever the concept allows it.
  2. **A "use client" component with real local state**, only when something about the concept can't work executed straight inside this app's own render (needs its own real routing/params/thrown error that would conflict with this app's own). Rare for backend topics compared to the sibling's frontend ones ‚Äî most backend concepts (event loop, middleware, routing, error handling) CAN be proven with a real, live, server-side execution instead of a rebuilt stand-in. Reach for this only when pattern 1 genuinely doesn't apply.

## The Turbopack execFileSync gotcha (confirmed 2026-07-13)

Spawning a real example process from a Server Component runner via `child_process.execFileSync("node", [somePathString])` **fails `next build`** with `Module not found ... server relative imports are not implemented yet` ‚Äî Turbopack's build-time file tracer statically treats a literal/computed string in that array-argument position as a module specifier to resolve, regardless of whether the path is built with `path.join`, `__dirname`, `process.cwd()` concatenation, or even a bare relative filename. Confirmed directly across all of those variations ‚Äî every one broke the build the same way.

**The fix that works:** use `child_process.execSync("node event-loop-order.js", { cwd: ..., encoding: "utf-8" })` ‚Äî a single shell-command string plus a `cwd` option, not an array of path-shaped arguments. This sidesteps whatever heuristic Turbopack applies to `execFileSync`'s argument array. See `src/example-runners/NodeRuntimeFoundations/EventLoopOrderRunner.tsx` for the working reference implementation. Use this pattern for every future runner that needs to spawn a real process rather than rediscovering the same failure.

## The missing body text-color gotcha (confirmed 2026-07-13)

`src/app/globals.css` styled `body` with a font-family but never a default `color` ‚Äî so any text node without an explicit Tailwind `text-*` class (title/body/sublabel/muted, or a semantic accent) fell back to the browser's default black, invisible against this app's black background. Caught in the `LibuvPhasesDiagram` on the `what-is-nodejs` page, where a raw `{i + 1}.` and ` ‚Äî ` sat directly in a `<div>` with no color class. Fixed globally by adding `color: var(--color-body);` to the `body` rule ‚Äî this is the actual root-cause fix, not just patching the one diagram. **When writing any new bespoke diagram, every text-bearing element still needs an explicit color class anyway** (don't rely on inheritance as the primary plan) ‚Äî the body-level fix is a safety net, not a reason to skip explicit `text-body`/`text-muted`/accent classes on new markup.

## One mini-project PER SECTION, not one shared per topic (confirmed 2026-07-14)

Every distinct `StudySection` on a topic page needs its OWN separate, self-contained `examples/<Topic>/<SectionSpecificName>/` mini-project ‚Äî its own folder, own `package.json`, own server/script file containing ONLY the code that section actually demonstrates. **Do not** build one shared server.js (or one shared example project) and then split it across multiple sections via multiple demo scripts that each hit different routes on the same file ‚Äî that was the actual mistake made across all three of Stage B's first topics (caught directly by the user on "The Middleware Pipeline," where 3 sections covering 3 genuinely unrelated concepts ‚Äî pipeline order, an auth-check that only works in the right position, a forgotten next() hanging a request ‚Äî all lived in one shared `server.js`).

**Why this matters, beyond just tidiness:** a shared file across sections makes it unclear which code belongs to which lesson, and edits/fixes to one section's demo risk silently affecting another section's. Each section is a separate lesson and deserves a separate, fully self-contained mini-project, even if that means near-identical boilerplate (a `package.json`, an Express import, a `pathToFileURL` listen-guard) gets repeated across a topic's several mini-projects ‚Äî that repetition is the correct trade-off here, not premature duplication to avoid.

**How to apply going forward:** when planning a new topic, decide the sections FIRST, then build one mini-project per section from the start ‚Äî never build one broad example and carve demo scripts out of it after the fact.

## FilePointers must stay inside that topic's own `examples/` folder (confirmed 2026-07-13)

Every `filePointer`/`filePointers` entry on a topic page must resolve to a path inside that topic's own `examples/<Topic>/<SubDemo>/` mini-project ‚Äî **never** to a root-level file (root `package.json`, `package-lock.json`, `CLAUDE.md`, etc.), even when the root file is real and the claim being made is true. The teaching surface for every topic is its own self-contained mini-project, full stop. If a concept needs a real artifact to point at (a real dependency to show semver ranges, a real generated lockfile), build/install that artifact inside the example folder itself ‚Äî e.g. `examples/ModulesNpmPackageJson/CjsEsmInterop/package.json` declares a real dependency (`picocolors`, actually used in `main.js` to color its real output) specifically so its own `package-lock.json` could be pointed at directly, instead of reusing the root project's. Caught directly by the user on the "Modules, npm & package.json" topic pointing at root `package.json`/`package-lock.json` ‚Äî don't rediscover this.

## Root tooling must exclude `examples/` (confirmed 2026-07-13)

Root `tsconfig.json`'s `include: ["**/*.ts", "**/*.tsx", ...]` and `eslint.config.mjs` both glob broadly by default ‚Äî the first example with real `.ts` source files (`examples/TypeScriptWithNode/TypedBackendSetup/`) got pulled straight into the root app's own typecheck (under the ROOT app's `moduleResolution: "bundler"` settings, not that example's own `nodenext` + `rewriteRelativeImportExtensions` config), producing false errors that had nothing to do with the app itself. Fixed by adding `"examples"` to root `tsconfig.json`'s `exclude` array and `"examples/**"` to `eslint.config.mjs`'s `ignores`, alongside the existing sibling-project exclusions ‚Äî both already fixed, so this shouldn't recur, but if a *new* root tool ever gets added (a second linter, a different type-checker config, etc.), remember every `examples/<Topic>/<SubDemo>/` mini-project is self-contained with its own tooling and must stay out of the root app's scan.

## The same-process blocking-demo gotcha (confirmed 2026-07-14)

Proving "blocking the event loop freezes the whole server" with a single Node process acting as BOTH the HTTP client and the server (fetch() calling into its own http.createServer) has two real, confirmed traps:

1. **Racing two different routes (fire /blocking and /ping "concurrently" via Promise.all, measure which one is slow) is genuinely flaky** ‚Äî whichever request's TCP connection happens to get accepted and dispatched to the JS callback first isn't guaranteed to be the one issued first in code. Confirmed directly: out of 3 runs, one showed /ping resolving in 8ms even while /blocking's busy-loop should have still been running.
2. **A client-side setTimeout "head start" to force ordering doesn't work either** ‚Äî since client and server code run on the exact same single JS thread in a same-process demo, a client-side timer scheduled to fire in 30ms can itself get swallowed by the server's synchronous busy-loop if that loop starts running before the 30ms elapses (confirmed via added instrumentation: a timer set for 30ms didn't fire until t+316ms, right after a 300ms busy-loop that started at t+14ms finished ‚Äî the timer was blocked too, exactly like everything else).

**The fix that works, deterministically, across 8+ runs:** don't race two different routes ‚Äî fire TWO CONCURRENT requests to the SAME route and measure the real TOTAL time for both. Blocking work can't overlap with itself (total ‚âà 2x the per-request work, serialized); non-blocking work does overlap (total ‚âà 1x). This sidesteps the ordering race entirely, since it no longer matters which request's connection gets dispatched first. See `examples/AsyncPatternsOnServer/AsyncOnAServer/blocking-server-demo.js` for the working reference implementation.

## The ESM "run directly vs imported" check (confirmed 2026-07-14)

Every Express example server needs to both (a) export its `app` for a demo script to import and drive via ephemeral-port `fetch()` calls, and (b) actually call `.listen()` for real when run directly via `node server.js`. The standard CommonJS check (`require.main === module`) doesn't exist in ESM ‚Äî the naive-looking ESM equivalent, comparing `import.meta.url` against `` `file://${process.argv[1]}` ``, is a real, confirmed bug: `process.argv[1]` is frequently a RELATIVE path (e.g. `"server.js"`), while `import.meta.url` is always an absolute `file://` URL, so the string comparison silently never matches ‚Äî `.listen()` never runs, the process exits immediately with zero output, and it looks like nothing is wrong until you actually try to hit the server (confirmed directly: this exact bug shipped once in `examples/ExpressAppRouting/BasicExpressApp/server.js` before being caught).

**The real fix:** `import { pathToFileURL } from "node:url"; if (import.meta.url === pathToFileURL(process.argv[1]).href) { app.listen(...) }` ‚Äî `pathToFileURL` correctly resolves a relative-or-absolute `argv[1]` into the same absolute-URL shape as `import.meta.url`, regardless of how the script was invoked. Use this pattern for every future Express server example that needs both a real standalone `node server.js` entry point AND an importable `app` for a demo script.

## The Express 4 async footgun crashes the process, it doesn't just hang (confirmed 2026-07-15)

Conventional wisdom (and this project's own earlier curriculum summary) described the pre-Express-5 async-handler footgun as "the request hangs forever." Verified for real on the pinned Node version in this project (Node v22.22.3): an Express 4 async route handler that throws, with no try/catch and no next(err), does NOT just hang that one request ‚Äî it produces an **unhandled promise rejection**, and modern Node.js (v15+) treats that as a fatal error by default, **crashing the entire process**. Confirmed deterministically across multiple runs: `spawn`ing a real Express 4 server as a separate process and hitting the offending route ends with that process exiting with code 1 (a `fetch failed: other side closed` `SocketError` on the client side, from the connection being yanked out from under it) ‚Äî taking down every other in-flight request on that same server, not just the one that triggered it. See `examples/ErrorHandlingExpress/LegacyTryCatch/demo.js` for the working reference implementation (it spawns the server as a real child process specifically so there's a real process to observe dying, rather than trying to survive the crash in-process).

**Why this matters pedagogically:** this is a stronger, more accurate teaching point than "hangs forever" ‚Äî Express 5's automatic rejected-promise-forwarding isn't just closing a request-hanging edge case, it's preventing one bad async route from taking the ENTIRE server down. Use "crashes the whole process" as the framing going forward, not "hangs," whenever this topic comes up again.

## Wrong/anti-pattern code lives only in comments, never as real running code (confirmed 2026-07-15)

Caught directly by the user across three shipped Stage B examples (`MiddlewarePipeline/AuthOrderMatters`'s `/protected-wrong` route, `MiddlewarePipeline/ForgottenNext`'s entire premise, `ErrorHandlingExpress/LegacyTryCatch`'s `/no-trycatch` route): shipping a deliberately-broken/anti-pattern route as REAL, executing code in an example's `server.js` is wrong, even when the point of the section is to teach why that pattern is broken. The reasoning: every example's `server.js` doubles as reference material for code the user might genuinely reuse or pattern-match against when writing real production code later. A file that contains real, running "wrong" code ‚Äî even clearly commented as wrong ‚Äî risks becoming muscle memory precisely because it's real, syntactically valid, executed code sitting right next to the correct version.

**The fix, applied as a standing rule for every future topic, not just a one-time cleanup:** when a section's whole teaching point is "here's a mistake and why it's bad," the wrong/anti-pattern version is written ONLY as a comment block inside the file ‚Äî clearly labeled (e.g. "THE MISTAKE, shown only as a comment, never actually run in this project") ‚Äî never as a real route, real middleware, or any other code that Express/Node actually executes. The comment should still state the real, verified consequence of that mistake (crashes the process, hangs forever, silently does nothing) as a plain factual statement, because that fact was actually verified by running the broken version once during development ‚Äî it just doesn't ship as part of the example. The ONLY code that actually runs in the shipped example is the correct, production-worthy version ‚Äî the one the user should treat as "what I'd actually write." Demo scripts only ever exercise the correct, real, running routes.

**Verification discipline still applies to the comment's claims** ‚Äî before writing a comment asserting what the broken version does (hangs / crashes / silently succeeds), actually run it once, off to the side, and confirm the real behavior first. Don't write an unverified claim into a comment just because it's "only a comment" now.

**This also covers incompleteness, not just active bugs** ‚Äî caught a 4th instance the same session, on `ErrorHandlingExpress/AutomaticForwarding`: the shipped `/risky` route threw with no try/catch (correct ‚Äî that's the point of the section) but had NO error-handling middleware registered at all, silently relying on Express's own default handler (which leaks a raw stack trace to the client). That's not a route a real backend dev would ship either, even though nothing about it looks like an obvious "bug" the way a forgotten `next()` does. Fixed by adding a real error middleware to the shipped code, with the default-handler fallback behavior (verified: a full HTML page with the stack trace, since `NODE_ENV` isn't `"production"`) described only in a comment. **When reviewing a section's shipped code against this rule, check for what's silently MISSING (a middleware nobody registered, a check nobody wrote) just as much as for what's ACTIVELY wrong** ‚Äî both count as "not what a real backend dev would ship."

## Non-negotiable rules

- **Never fabricate behavior.** If a topic makes a claim about how Node/Express/a library behaves (error propagation, middleware order, a status code, a header, an ordering guarantee), verify it by actually running the code first ‚Äî don't assert from memory or training data. Library/runtime behavior changes across versions, and this project already found one genuinely surprising, verified-for-real gotcha in its first topic (setTimeout vs setImmediate ordering at the top level of a script is NOT guaranteed ‚Äî see the `what-is-nodejs` topic page).
- **No fake stand-ins where the topic is genuinely about a real tool.** If the topic is about validation, use a real validator; if it's about a database, connect to a real one (can be local/throwaway, but real). Don't simulate what a real dependency would do.
- **Runner output must match the example it claims to demonstrate, exactly.** If the example prints/returns N things, the runner's shown output must show all N ‚Äî no silently trimming, no adding anything that isn't really there.
- **Keep pinned-version/gotcha notes in `CLAUDE.md` up to date** if a dependency version or tooling behavior turns out to break something in this environment ‚Äî don't silently work around it without recording why, the way the Turbopack gotcha above is now recorded.

## Verification discipline

Before writing prose that asserts "Express does X" or "Node does Y" as fact, actually run it and look at the real output first ‚Äî including running it multiple times when a claim is about ordering/timing, since that's exactly the kind of claim that looks deterministic until you check (see the `what-is-nodejs` topic). Record confirmed gotchas here or in `CLAUDE.md` as they're found, not just in the moment they're discovered.
