# Roadmap

Owned by `skillCoFounderMentor.md`. Reference map, not a strict order — `current-state.md` says what's actually active right now. Edit this file if the plan itself changes; don't touch it just because a session happened.

**Stage A — Node.js & Backend Foundations**
What Node actually is (V8, libuv, the event loop, why "single-threaded but non-blocking" matters), npm/package.json/semver/scripts, **TypeScript with Node.js** (tsconfig for a backend, tsx/native type-stripping for dev vs `tsc` for prod builds — table-stakes in 2026, not optional polish), core modules (`fs`, `path`, `process`, `events`, streams/buffers basics, plus the Web-standard globals Node now ships natively — `fetch`, `Headers`, `FormData`, `structuredClone` — a real bridge from the fetch() already known from frontend work), async patterns reframed for a server (callbacks → promises → async/await, why blocking the event loop is a real production bug), debugging a running Node process.

**Stage B — Express Fundamentals**
**Reordered 2026-07-17: "API Testing Tools" is now Stage B topic 1**, ahead of App/router basics — moved so curl/Postman/Thunder Client are already familiar before every later Stage B (and beyond) topic starts including its own explicit Postman verification steps per section (see `build-conventions.md`'s Postman-guide entry). **API testing tools** (curl — the terminal workflow used throughout this project's own verification passes — plus Postman and Thunder Client as the GUI alternatives, and the plain-text `.http` request-file format both are built around), then app/router basics — **targeting Express 5**, npm's default and the Express Technical Committee's production-recommended release as of 2026 (Express 4 enters formal EOL no sooner than October 2026) — the middleware pipeline and why order matters (a top interview question), request/response objects, routing patterns (params/query/nested routers), error-handling middleware (Express 5's automatic rejected-promise-to-error-middleware forwarding is the modern default; the manual try/catch pattern it replaces is still worth knowing as legacy-codebase awareness), project structure/layering conventions, serving JSON/static assets, environment config (`dotenv`).

**Stage C — Data Layer**
**Decided 2026-07-15, no longer TBD: both PostgreSQL and MongoDB, taught in real depth side by side** — the user explicitly asked to learn both rather than picking one, since a full Stage C pass is a good place to genuinely master the SQL/NoSQL divide rather than just gesture at it. SQL vs NoSQL decision-making (the real trade-offs, not just definitions), connecting a real PostgreSQL database via Prisma AND a real MongoDB database via Mongoose (same CRUD operations, compared side by side), migrations & schema design (Prisma Migrate's versioned relational schema changes, contrasted with MongoDB's embedding-vs-referencing document design decisions), transactions & N+1/performance gotchas in both a relational store (joins) and a document store (Mongo's own multi-document transactions and embedding-to-avoid-joins strategy). Each Stage C topic is expected to have real sections/mini-projects for BOTH databases, not just one — same one-mini-project-per-section convention as every other stage. **Added 2026-07-19: "Indexing & Search Performance"** — B-tree indexes + `EXPLAIN ANALYZE` on real Postgres (measuring the actual before/after query-time gap an index makes, the same measured-not-asserted discipline as the pagination/N+1 demos), plus Postgres full-text search (`tsvector`/`tsquery`, GIN indexes) — buildable with zero new infra since it's the same Coolify Postgres already connected. Elasticsearch covered as a conceptual comparison only, not installed — bigger infra lift for a narrower payoff at this project's scope.

**Stage D — API Design & Real-World Concerns**
REST conventions, status codes, URL-path versioning (still the 2026 gold standard — explicit, cache-friendly), **cursor vs offset pagination** (offset collapses at scale — `OFFSET 100000` forces a full scan-and-discard), **idempotency keys** (the Stripe/PayPal pattern that makes retries safe), input validation (zod/joi), auth (sessions vs JWT, password hashing with bcrypt), authorization/RBAC basics, CORS — the literal handshake with a frontend team, high-value given the existing frontend background — **OWASP API Security Top 10 framing** for the security topic (broken access control/BOLA is the single most common real-world API vuln, found in ~88% of audited Node APIs; injection at the API boundary; helmet for security misconfiguration), rate limiting. **"File Uploads & Blob Storage" made a real topic 2026-07-19** — previously just named in this prose without ever getting a `curriculum.ts` card, a real gap caught while triaging a system-design course syllabus against this project. The real pattern: presigned S3 upload URLs so the client uploads directly to blob storage rather than routing file bytes through Express, with a CDN in front for serving — this is the actual production pattern interviewers expect, not `multer`-to-local-disk.

**Stage E — Testing, Tooling & Production Readiness**
Unit + integration testing (Jest/Vitest + Supertest), linting/formatting, logging (pino/winston), error tracking, secrets/environment management, containerization basics (Docker), deployment — **decided 2026-07-20: the user's own existing self-hosted Coolify instance** (already running this project's Postgres + MinIO), not Render/Railway/Fly, specifically to reuse real infra already paid for — **debugging & memory profiling** (heap snapshots + comparison-based leak hunting — dangling event listeners and unbounded caches are the two classic culprits — `--inspect`, Clinic.js), caching (Redis) and scaling — sharpened to the real, clean distinction between the **cluster module** (separate processes, fault-isolated, right tool for I/O-bound HTTP load) and **worker_threads** (shared memory via `SharedArrayBuffer`, right tool for CPU-bound work only) — plus PM2 and load balancers. **Added 2026-07-19:** a real distributed lock built on the same Redis instance as the caching work (`SET NX PX` or a small library) — prevents double-processing, a genuine senior-level Redis pattern — folded into "Caching & Scaling" rather than given its own topic since it needs no new infra.

**Stage F — Advanced & Interview Prep**
WebSockets/real-time (Socket.io) — **the real-time progression sharpened 2026-07-19 to name polling → SSE (Server-Sent Events, one-way server-to-client over plain HTTP) → WebSockets (two-way) explicitly**, so the topic teaches when you'd reach for the simpler option instead of defaulting to WebSockets — **background jobs & message queues made concrete** — BullMQ + Redis is the de-facto 2026 standard for Node background job processing, contrasted with RabbitMQ/Kafka which solve a genuinely different problem (cross-service event streaming, not just "bigger queues") — microservices vs monolith tradeoffs, **Beyond REST: GraphQL & tRPC awareness** (when a team reaches for GraphQL — aggregating multiple resources for varied clients — or tRPC — end-to-end type safety with a TypeScript frontend, directly relevant given the existing Next.js background — awareness-level, not full depth), observability (metrics/tracing basics), backend system design interview drills (pagination/idempotency/rate-limiting choices are real discussion anchors here), live-coding a REST API under time pressure, behavioral prep narrating this project, and explicitly how backend and frontend teams collaborate (API contracts, OpenAPI/Swagger, mocking a backend for a frontend dev) — using the existing frontend background as a real interview differentiator. **Added 2026-07-19: named framing lenses for the system design drills topic** — CAP theorem (mapped onto the real Postgres-vs-Mongo tradeoffs already built in Stage C, not taught from scratch), sharding and consistent hashing (shard key choice, hash-ring structure, cross-shard cost), and the standard latency-numbers-every-engineer-should-know table — all covered as interview talking points, not hands-on builds, since they're genuinely infra/platform-team territory rather than something a Node/Express app builds itself.

## 2026 web-research pass (2026-07-13)

Ran a research pass (web search across ~10 queries: general Node/Express roadmaps, current interview-question surveys, Express 4-vs-5 release status, TypeScript backend setup, memory-leak debugging, REST idempotency/pagination/versioning, cluster-vs-worker_threads, OWASP API security, background job queues, GraphQL/tRPC-vs-REST relevance) to check this roadmap against current standards, the same kind of gap-analysis pass the sibling JavaScript project ran on its own curriculum 2026-07-10.

**Result: the overall shape and stage ordering held up well** — no stage needed restructuring. Real gaps found and folded into the stage descriptions above (all now reflected in `src/data/curriculum.ts` too):
1. **No explicit TypeScript-with-Node topic existed**, despite this project's own name being "Node + Express JS/TS" and TypeScript now being described as "table stakes" for any serious 2026 Node project — added to Stage A.
2. **Express version wasn't pinned as a decision.** Express 5 is npm's default and the TC's production-recommended release now — made explicit rather than only implied via the "pre-Express-5 footgun" phrasing, which read like Express 5 was a future/edge case rather than the actual target.
3. **Pagination and idempotency keys weren't named**, despite being concretely, repeatedly cited as 2026 production-reference material (Stripe/PayPal/GitHub-style idempotency; cursor-vs-offset pagination) — added to Stage D's REST topic.
4. **Security topic wasn't framed against OWASP API Security Top 10**, the standard reference lens interviewers actually use (broken access control/BOLA named as the single most common real API vuln in current sources) — reframed Stage D's security topic around it.
5. **No explicit memory-leak/heap-snapshot debugging topic**, despite this being named directly as a senior-interview-relevant skill distinct from generic logging — added to Stage E.
6. **Cluster vs worker_threads wasn't a named, contrasted pair** — Stage E's caching/scaling topic only said "clustering" vaguely. Sharpened to name both and the real distinction (process isolation + fault tolerance vs shared-memory CPU work).
7. **Background jobs/queues was only prose in this file, with no matching topic card in `src/data/curriculum.ts` at all** — a real gap between the plan and what actually shipped to the roadmap page, not just a content gap. Added as its own Stage F topic, made concrete (BullMQ + Redis, contrasted with RabbitMQ/Kafka).
8. **No awareness-level mention of GraphQL/tRPC** as alternatives to REST — added as a lightweight Stage F topic, deliberately not full-depth, with tRPC specifically flagged as relevant given the existing Next.js/TypeScript frontend background.

**Deliberately NOT added:** GraphQL/tRPC as full curriculum depth (REST remains what ~93% of teams actually ship, per research — this project's core is rightly REST-first); a dedicated gRPC topic (niche for this project's stated goals); TC39/framework-tied stage additions of the kind the JS sibling project also explicitly skipped (that project passed on TS decorators for the same reason — still shifting, framework-tied).
