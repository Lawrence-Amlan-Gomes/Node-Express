# Roadmap

Owned by `skillCoFounderMentor.md`. Reference map, not a strict order — `current-state.md` says what's actually active right now. Edit this file if the plan itself changes; don't touch it just because a session happened.

**Stage A — Node.js & Backend Foundations**
What Node actually is (V8, libuv, the event loop, why "single-threaded but non-blocking" matters), npm/package.json/semver/scripts, **TypeScript with Node.js** (tsconfig for a backend, tsx/native type-stripping for dev vs `tsc` for prod builds — table-stakes in 2026, not optional polish), core modules (`fs`, `path`, `process`, `events`, streams/buffers basics, plus the Web-standard globals Node now ships natively — `fetch`, `Headers`, `FormData`, `structuredClone` — a real bridge from the fetch() already known from frontend work), async patterns reframed for a server (callbacks → promises → async/await, why blocking the event loop is a real production bug), debugging a running Node process.

**Stage B — Express Fundamentals**
App/router basics — **targeting Express 5**, npm's default and the Express Technical Committee's production-recommended release as of 2026 (Express 4 enters formal EOL no sooner than October 2026) — the middleware pipeline and why order matters (a top interview question), **API testing tools** (curl — the terminal workflow already used throughout this project's own verification passes — plus Postman and Thunder Client as the GUI alternatives, and the plain-text `.http` request-file format both are built around; added 2026-07-14 after the user directly asked whether supporting tooling like this was covered, a real gap the roadmap hadn't named), request/response objects, routing patterns (params/query/nested routers), error-handling middleware (Express 5's automatic rejected-promise-to-error-middleware forwarding is the modern default; the manual try/catch pattern it replaces is still worth knowing as legacy-codebase awareness), project structure/layering conventions, serving JSON/static assets, environment config (`dotenv`).

**Stage C — Data Layer**
**Decided 2026-07-15, no longer TBD: both PostgreSQL and MongoDB, taught in real depth side by side** — the user explicitly asked to learn both rather than picking one, since a full Stage C pass is a good place to genuinely master the SQL/NoSQL divide rather than just gesture at it. SQL vs NoSQL decision-making (the real trade-offs, not just definitions), connecting a real PostgreSQL database via Prisma AND a real MongoDB database via Mongoose (same CRUD operations, compared side by side), migrations & schema design (Prisma Migrate's versioned relational schema changes, contrasted with MongoDB's embedding-vs-referencing document design decisions), transactions & N+1/performance gotchas in both a relational store (joins) and a document store (Mongo's own multi-document transactions and embedding-to-avoid-joins strategy). Each Stage C topic is expected to have real sections/mini-projects for BOTH databases, not just one — same one-mini-project-per-section convention as every other stage.

**Stage D — API Design & Real-World Concerns**
REST conventions, status codes, URL-path versioning (still the 2026 gold standard — explicit, cache-friendly), **cursor vs offset pagination** (offset collapses at scale — `OFFSET 100000` forces a full scan-and-discard), **idempotency keys** (the Stripe/PayPal pattern that makes retries safe), input validation (zod/joi), auth (sessions vs JWT, password hashing with bcrypt), authorization/RBAC basics, CORS — the literal handshake with a frontend team, high-value given the existing frontend background — **OWASP API Security Top 10 framing** for the security topic (broken access control/BOLA is the single most common real-world API vuln, found in ~88% of audited Node APIs; injection at the API boundary; helmet for security misconfiguration), rate limiting, file uploads.

**Stage E — Testing, Tooling & Production Readiness**
Unit + integration testing (Jest/Vitest + Supertest), linting/formatting, logging (pino/winston), error tracking, secrets/environment management, containerization basics (Docker), deployment (Render/Railway/Fly/VPS — TBD together), **debugging & memory profiling** (heap snapshots + comparison-based leak hunting — dangling event listeners and unbounded caches are the two classic culprits — `--inspect`, Clinic.js), caching (Redis) and scaling — sharpened to the real, clean distinction between the **cluster module** (separate processes, fault-isolated, right tool for I/O-bound HTTP load) and **worker_threads** (shared memory via `SharedArrayBuffer`, right tool for CPU-bound work only) — plus PM2 and load balancers.

**Stage F — Advanced & Interview Prep**
WebSockets/real-time (Socket.io), **background jobs & message queues made concrete** — BullMQ + Redis is the de-facto 2026 standard for Node background job processing, contrasted with RabbitMQ/Kafka which solve a genuinely different problem (cross-service event streaming, not just "bigger queues") — microservices vs monolith tradeoffs, **Beyond REST: GraphQL & tRPC awareness** (when a team reaches for GraphQL — aggregating multiple resources for varied clients — or tRPC — end-to-end type safety with a TypeScript frontend, directly relevant given the existing Next.js background — awareness-level, not full depth), observability (metrics/tracing basics), backend system design interview drills (pagination/idempotency/rate-limiting choices are real discussion anchors here), live-coding a REST API under time pressure, behavioral prep narrating this project, and explicitly how backend and frontend teams collaborate (API contracts, OpenAPI/Swagger, mocking a backend for a frontend dev) — using the existing frontend background as a real interview differentiator.

**All five Stage A topics are now built and mastered** (2026-07-14): "What is Node.js, really" (V8/libuv layering, setTimeout-vs-setImmediate non-determinism), "Modules, npm & package.json" (CJS/ESM interop, including the require()-can-now-load-sync-ESM behavior confirmed on Node 22.12+), "TypeScript with Node.js" (real typed project, native strip-types vs tsx vs tsc, the type-stripping-≠-type-checking gap), "Core Modules" (path/fs/Buffer/structuredClone/events/process/Headers/FormData/fetch tour, including a real local http server), and "Async Patterns on a Server" (callback/promise/async-await progression, plus a measured, real blocking-vs-non-blocking demo).

**All five Stage B topics are now built and mastered** (2026-07-15): "App & Router Basics (Express 5)", "The Middleware Pipeline", "API Testing Tools: curl, Postman & Thunder Client", "Error Handling in Express" (Express 5 automatic forwarding, plus a real Express 4 install proving the legacy async footgun crashes the whole process on modern Node, not just hangs), and "Project Structure & Config" (routes/controllers layering, real `.env`/dotenv config).

**All four Stage C topics are now built and mastered** (2026-07-16): "SQL vs NoSQL" (real SQLite tables+JOIN vs embedded documents), "Connecting Real Databases: PostgreSQL & MongoDB" (real Prisma+Postgres and Mongoose+Mongo CRUD against the user's own real Coolify Postgres server and MongoDB Atlas cluster — not local throwaway databases), "Migrations & Schema Design" (two real, applied Prisma Migrate migrations, plus a real MongoDB embed-vs-reference comparison), and "Transactions & Query Performance" (real rollback proofs on both databases, a measured, real N+1 query-count fix). Real environment decision made this stage: connect to real shared infrastructure the user already has, rather than installing local Docker/Postgres/MongoDB — see `build-conventions.md` for the isolation conventions this required (dedicated schemas per Prisma mini-project, dedicated Mongo collections).

**All four Stage D topics are now built and mastered** (2026-07-16): "REST Conventions & Validation" (status codes, a measured ~300x real cursor-vs-offset pagination gap, idempotency keys, zod), "Authentication Patterns" (bcrypt, sessions, JWTs), "CORS — the Frontend Handshake" (the first topic needing a real headless browser via Playwright, since CORS is browser-enforced, not Node-enforced), and "OWASP API Security & Rate Limiting" (BOLA, SQL injection, rate limiting, helmet).

Next unbuilt: **Stage E, topic 1 — Testing an Express App** (Jest/Vitest + Supertest).

## 2026 web-research pass (2026-07-13)

Ran a research pass (web search across ~10 queries: general Node/Express roadmaps, current interview-question surveys, Express 4-vs-5 release status, TypeScript backend setup, memory-leak debugging, REST idempotency/pagination/versioning, cluster-vs-worker_threads, OWASP API security, background job queues, GraphQL/tRPC-vs-REST relevance) to check this roadmap against current standards, the same kind of gap-analysis pass the sibling JavaScript project ran on its own curriculum 2026-07-10.

**Result: the overall shape and stage ordering held up well** — no stage needed restructuring. Real gaps found and folded into the stage descriptions above (all now reflected in `src/data/curriculum.ts` too):
1. **No explicit TypeScript-with-Node topic existed**, despite this project's own name being "Node + Express JS/TS" and TypeScript now being described as "table stakes" for any serious 2026 Node project — added to Stage A.
2. **Express version wasn't pinned as a decision.** Express 5 is npm's default and the TC's production-recommended release now — made explicit rather than only implied via the "pre-Express-5 footgun" phrasing, which read like Express 5 was a future/edge case rather than the actual target.
3. **Pagination and idempotency keys weren't named**, despite being concretely, repeatedly cited as 2026 production-reference material (Stripe/PayPal/GitHub-style idempotency; cursor-vs-offset pagination) — added to Stage D's REST topic.
4. **Security topic wasn't framed against OWASP API Security Top 10**, the standard reference lens interviewers actually use (broken access control/BOLA named as the single most common real API vuln in current sources) — reframed Stage D's security topic around it.
5. **No explicit memory-leak/heap-snapshot debugging topic**, despite this being named directly as a senior-interview-relevant skill distinct from generic logging — added to Stage E.
6. **Cluster vs worker_threads wasn't a named, contrasted pair** — Stage E's caching/scaling topic only said "clustering" vaguely. Sharpened to name both and the real distinction (process isolation + fault tolerance vs shared-memory CPU work).
7. **Background jobs/queues was only prose in this file, with no matching topic card in `src/data/curriculum.ts` at all** — a real gap between the plan and what actually shipped to the roadmap page, not just a content gap. Added as its own Stage F topic, made concrete (BullMQ + Redis, contrasted with RabbitMQ/Kafka).
8. **No awareness-level mention of GraphQL/tRPC** as alternatives to REST — added as a lightweight Stage F topic, deliberately not full-depth, with tRPC specifically flagged as relevant given the existing Next.js/TypeScript frontend background.

**Deliberately NOT added:** GraphQL/tRPC as full curriculum depth (REST remains what ~93% of teams actually ship, per research — this project's core is rightly REST-first); a dedicated gRPC topic (niche for this project's stated goals); TC39/framework-tied stage additions of the kind the JS sibling project also explicitly skipped (that project passed on TS decorators for the same reason — still shifting, framework-tied).
