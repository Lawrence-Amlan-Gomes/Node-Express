## The detail-rebuild pass — active initiative, not yet finished

At the user's explicit request, every already-built topic is being rewritten to a much more detailed, easier-vocabulary, more visually self-explanatory standard, one topic at a time, in build order. Full rules live in `co-founder/build-conventions.md`'s "detail-rebuild pass" entry and its follow-on entries — **read that before touching any topic**. Summary of the standard:

1. **Prose** uses short, plain-vocabulary sentences with visible structure — bold numbered sub-headers via a `ConceptBreakdown` component, closed out with a `Callout` "The Bottom Line" box. `StudySection` (`src/components/StudyPage.tsx`) supports a `body?: ReactNode` field for this now, alongside the older `paragraphs: string[]` (not-yet-rebuilt topics still use `paragraphs`).
2. **Every non-Interview-Angle section needs its own bespoke diagram** — a live demo or a `ComparisonCard` does NOT substitute for one. Interview Angle sections are exempt (their `ComparisonCard` recap is the accepted format there).
3. **Bespoke diagrams must be self-explanatory** — every element gets its own plain-English caption, not just a label.
4. **Example code gets a comment on (or directly above) nearly every line — one comment per STATEMENT, never one comment governing a whole run of several statements.** Caught drifting back into batching mid-pass once already; the test to apply: could you delete one code line without needing to edit the comment above a DIFFERENT line? If not, split it.
5. **Two color/style fixes, apply by default on every topic:** use `text-sublabel` (`#a3a3a3`) for secondary/muted diagram captions, never `text-muted` (`#444444`, near-invisible on this app's black background); keep colored diagram box backgrounds at very low opacity (`bg-<color>-500/3`, not `/10`) so they read as near-black with just a colored border/label.
6. **Also check every topic against OLDER pre-existing conventions while rebuilding it**, not just the new ones — several early topics (built before those rules existed) still violated "one mini-project per section" and "every section needs its own live demo." Topics 4 and 5 of Stage A both needed a real folder split to fix this; don't assume a topic complies just because a newer topic would.
7. **From Stage B topic 2 onward, every section whose example is a real Express server on a real FIXED port** (not the ephemeral `listen(0,...)` used only for this app's own in-page live-demo proof) **gets a `PostmanCheck` "Try It Yourself in Postman" box** (`src/components/PostmanCheck.tsx`, rendered via `StudySection.postmanCheck`, at the very end of a section after the live demo). Every value in it must be genuinely verified — start the server standalone and curl/test it directly, never reuse ephemeral-port output blindly, never invent a response. The component takes `folderPath` (renders a literal, copy-pasteable `cd "/full/absolute/path"` as its own first step — the user wants exact terminal commands, not "run this yourself"), `runCommand`, `runPort`, and `steps`. **Stage B topic 1 ("API Testing Tools") is exempt** — it already teaches Postman directly, a guide box there would be redundant. Topics with no real fixed-port Express server (pure scripts, SQL/Mongo comparisons) correctly get no `PostmanCheck` at all — don't force one where there's nothing to hit over HTTP.
8. **"The Bottom Line" `Callout` must be a real, standalone concept summary — never "the demo below proves X."** That sentence is still real and belongs on the page, but as a separate `<p>` placed AFTER the Callout, never merged into or before it. Standing shape: `ConceptBreakdown`/prose → `Callout` (concept summary) → a short `<p>` describing what the demo proves. Caught 2026-07-17 on "App & Router Basics," found to have regressed into 3 more already-shipped sections the same session — re-check this ordering opportunistically on older rebuilt topics if it comes up again, the same way comment-batching regressions get proactively re-audited.
9. **The "outer comment doesn't cover nested lines" trap** — an `if`-guard, a named helper function, or a `.catch()/.finally()` callback getting one good comment on its own header line does NOT excuse the individual statements nested inside it from the same per-statement rule. `app.listen(...)` calls inside an ESM guard block and `console.log(...)` calls (especially the LAST one in a script, or ones inside a small helper) are the most commonly missed cases. Re-apply the per-statement rule to every line inside any block statement, not just the block's own header.
